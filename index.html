Traditional ETL pipelines rely on complex, fragile Regular Expressions to clean data. But what happens when the data is completely unpredictable?

This project replaces legacy cleaning entirely with an AI-driven parser (Llama 3.1 via Groq) — it understands context, corrects malformed records, and normalises data with human-like reasoning, all orchestrated inside a robust, idempotent Apache Airflow DAG.

Why This Project Exists
01 / 03 🧠
AI-Driven Cleaning

Uses Llama 3.1 to handle ambiguous normalisation that regex simply cannot catch. What would take 500 lines of complex regex is replaced by a single intelligent prompt. Adapts to new data flaws instantly — no rule updates needed.

02 / 03 🔁
Idempotent Loads

Pipeline safely reruns without duplicating records. Uses Postgres ON CONFLICT DO UPDATE — running the DAG 100 times yields the exact same database state as running it once. Production-safe by design.

03 / 03 🐳
100% Containerised

Full stack — Airflow, Postgres, all scripts — launches in under 2 minutes with Docker. make start is all you need. Zero environment setup headaches. Fully reproducible on any machine.

Why This Architecture · For Engineering Managers
Reason 01
LLM over Rule-Engines

Rules break when upstream systems change. LLMs adapt. What would take 500 lines of complex regex is replaced by a single intelligent prompt. When a new data flaw appears, there is no code to update — the model handles it.

Reason 02
Resilience Built-In

The pipeline features exponential backoff and 3× retries for every LLM network call. If Groq goes down entirely, it gracefully falls back to raw data — the pipeline never hard-crashes. Designed for production from day one.

Reason 03
Idempotency Guaranteed

Using Postgres ON CONFLICT DO UPDATE, running the DAG 100 times yields the exact same state as running it once. Reprocessing a day's data, recovering from failures, backfilling — all safe, all predictable.
